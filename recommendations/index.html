Recommendations

Introduction

I really like programming, and everything surrounding programming. I listen to youtube videos and read a lot of articles and books, and though that I could share those who have affected me the most and in what way.

List

Name Description
Breaking systems into modules A good overview of how you can describe modules in terms of incoming/public/private dependencies, public abstraction and private implemantation. It argues that making the smallest possible modules isn't always desirable because then you increase complexity in the dependency graph between modules.
All advice comes from a context Taking rules to their logical extremes seldom leads to good code. The article brings up the problems with fads and how most advice doesn't apply 100% of the time. The question remains on how to teach junior programmers this, since it can take a lot of time to internalize very loose rules.
Data as a mediator between computation and state Mutations can be described as actions (data) which can make systems easier to test, and can make it easier to introduce changes to cross-cutting concerns.
Deconstructing SOLID design principles SOLID principles are mostly relevant on system boundaries, yet people advocate for using them everywhere.
Deconstructing the "Unix philosophy" The article covers the advantages of "text as a universal interface" and the composability of unix commands. It also brings up that the division of commands may make sense from a developer point of view, but interactive users are more task-oriented. They don't want to have to cobble together often non-reusable shell scripts for the kind of tasks they are interested in. It also contrasts "text as a universal interface" with "object as a universal interface" from PowerShell.
Design duality and the expression problem APIs can be categorized based on if they guarantee extensibility through additional variants, operations or neither. Whatever extensibility category is promised disables adding internal changes in that category without breaking contracts. ADTs makes the fewest amount of guarantees and therefor enables the greatest internal flexibility if that is desired.
Epigrams in Programming Some fun ideas to think about. My favorite is "A LISP programmer knows the value of everything, but the cost of nothing." It highlights that building extensible solutions in programming often involves disregarding runtime efficiency to a certain degree. A very static program can be written very efficiently, but it may not be easy to evolve it in the direction that stakeholders need it to without rewrites.
Patterns vs. Higher Order Functions Patterns and higher-order functions are two concepts used to describe the same underlying concept in functional programming and object-oriented programming.
Parse, don't validate Bake assumptions into your types to make invalid programs impossible to express. e.g. Rather than send a list around, and verifying that it contains at least one element as a pre-condition everywhere, create a at-least-one element list, and use it everywhere that needs the pre-condition.
Testing at the boundaries The article brings up that when unit testing, it doesn't usually make sense to test every class in complete isolation. It is also critical of how the TDD concept has been warped over the years and that it started out as much more reasonable.
Testing, induction, and mocks A critical critique of mocking and how it should be a last resort. Code should be structured to make mocking next to unnecessary in unit tests.
Types are the basic tool of software design Functions with a high-level of generality can sometimes make the implementation hard to write incorrectly, since an incorrect implementation would have to make assumptions that weren't part of the signature.
Designing Data-Intensive Applications This book goes over a lot of different alternatives to storing and managing data, and discussing their trade-offs in large systems with high performance requirements.
Domain-Driven Design: Tackling Complexity in the Heart of Software This book opened my eyes to how programming fits into organizations and their needs. Not as something alien to be cornered off from the organization, but as something that should constantly be involved. It showed me that creativity and empathy can be really important for a programmer.
Effective Java A book that really highlights that to get the most from a programming languages you need to consider its' features as part of greater whole. Some features are useful more often than others, and some have even become obsolete over time. Some patterns aren't explicit language features, but should be treated as if they were.
Extreme Programming Explained: Embrace Change, Second Edition This book gave me insight into how it is to truly work iteratively, and what a great concept that is. It helped me take one step at a time when coding, instead of attempting to make enormous changes that were never really done and slow to test. It also opened my eyes to why we write user stories, and how important users really are.
Large-Scale C++ Software Design This book opened my eyes to how seemingly small design changes can reduce compile times for C++. It also taught me the importance of designing maintainable APIs.
Java Concurrency in Practice A really great overview about the concurrency primitives available in Java. It also shows the advantages of using immutable data structures for concurrency instead of sharing mutable state.
Practical Object-Oriented Design, An Agile Primer Using Ruby This book presents a good example of how to structure object-oriented code, and how to deal with testing in dynamic languages.
Programming in Lua My first introduction to a dynamic programming languages. I was moved by how effortlessly the author introduced concepts and built on top of those concepts without bringing up to much at the same time. It also got me to appreciate the implementation simplicity of Lua.
Mythical Man-Month, The: Essays on Software Engineering A good historical piece of issues that plagued programming in 19th century, and that mostly still exists today but in a different shape. The second system effect still lead to tools today that bite off more than they can chew.
The Pragmatic Programmer A goodie bag of advice around software development that every software developer should read.
99 Bottles of OOP A great book focused around teaching object-oriented design, how it relates to TDD, and how it's better to duplicate code than use the wrong abstraction.
Working Effectively with Legacy Code A great cookbook for ways to improve and make changes to legacy systems. Highlights the importance of automated testing to improve quality over time.
A Philosophy of Software Design A code quality book that focuses a lot on improving the interfaces at system boundaries. Has an interesting take on the value of comments.
David Schmitz - 10 Tips for failing badly at Microservices A funny presentation about how certain decisions can hurt more than others in preventing successful adoption of microservices.
YOW! 2013 - The SOLID Design Principles Deconstructed The SOLID design principles as we know them aren't that useful. The author breaks them down and proposes alternatives.
Integrated Tests Are A Scam A somewhat hyperbolic talk about how integration testing between systems often break, and should in many cases be replaced with contract testing.
CppCon 2014 - Data-Oriented Design and C++ A very though-provoking presentation about data-oriented design and how it might be useful for high-performance application.
Ruby on Rails demo A revolutionary presentation at the time about convention-over-configuration and how to quickly develop seemingly simple applications such as CRUD and blogs.
OOPSLA Conference: Growing a Language A presentation about how to develop the Java programming language in the future. It argues that an overly small language may not be that useful to many people.
Why Isn't Functional Programming the Norm? A presentation about the history of object-oriented programming and functional programming and how they diverged.
Every Clojure Talk Ever This is a pretty funny talk about hype-driven development in Clojure.
Building a psychologically safe workplace It's important that people feel safe that they can bring up issues. If not for moral reasons, then for the fact that otherwise people don't bring up problems. Those problems sometimes become really big, and we really want to solve them before that happens.
Programming is terrible — Lessons learned from a life wasted A cynical presentation about a lot of different programming aspects, mostly related to social issues.
Functional Design Patterns A very good presentation about functional programming concepts targeted towards programmers who know object-oriented programming.
